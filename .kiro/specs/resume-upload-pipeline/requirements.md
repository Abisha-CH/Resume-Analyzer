# Requirements Document

## Introduction

The **Resume Upload Pipeline** feature enables authenticated users to upload their resume files (PDF or DOCX) through a functional drag-and-drop interface in the dashboard. The uploaded file is stored in Supabase Storage under a collision-safe path, and a corresponding metadata record is inserted into the `resumes` database table with an initial status of `"pending"`. The feature replaces the existing static placeholder at `app/(dashboard)/upload/page.tsx` with a fully operational client component backed by a secure Next.js Route Handler.

All file handling, authentication verification, storage operations, and database writes are performed server-side. The browser only transmits the file via a `multipart/form-data` POST request to the Next.js API route — no Supabase credentials or service-role keys are ever exposed to the client.

---

## Glossary

- **Upload_API**: The Next.js Route Handler at `POST /api/resumes/upload` responsible for authenticating the request, validating the file, uploading it to storage, and inserting the database record.
- **Upload_UI**: The functional Client Component that replaces `app/(dashboard)/upload/page.tsx`, providing drag-and-drop file selection, upload progress feedback, success state, and error state.
- **Resume_File**: A user-supplied file in PDF or DOCX format, at most 5 MB in size, representing the user's resume.
- **Resume_Record**: A row in the `resumes` database table storing metadata about an uploaded Resume_File.
- **Resume_ID**: A unique identifier for a Resume_Record, generated server-side using `crypto.randomUUID()` (Node.js built-in; no additional dependency required).
- **Storage_Path**: The location of a Resume_File within the `"resumes"` Supabase Storage bucket, following the convention `{clerkUserId}/{resumeId}.{ext}`.
- **Clerk_User_ID**: The string identifier (e.g. `"user_2abc…"`) assigned by Clerk to an authenticated user. In this application, `users.id` (the primary key of the `users` table) stores the Clerk User ID directly — `users.clerkId` is a redundant duplicate of the same value. See the [User Identity Design Clarification](#user-identity-design-clarification) section.
- **Service_Client**: The Supabase client created via `createServiceClient()` from `lib/supabase/server.ts`, initialized with the `SUPABASE_SERVICE_ROLE_KEY`. Used exclusively in server-side code to bypass Row Level Security for trusted operations.
- **Anon_Client**: The Supabase client created via `createClient()` from `lib/supabase/client.ts`, initialized with the public anon key. Used in browser-side Client Components for non-privileged operations. Must never be used for resume storage writes.
- **FormData**: The HTTP `multipart/form-data` encoding used to transmit the Resume_File from the browser to the Upload_API, with the file attached under the field name `"file"`.
- **Signed_URL**: A time-limited, pre-authenticated URL generated by Supabase Storage that grants temporary read access to a private object.
- **MIME_Type**: The content type string carried in the HTTP request header for the uploaded file (e.g. `"application/pdf"` or `"application/vnd.openxmlformats-officedocument.wordprocessingml.document"`).
- **pending**: The initial `resume_status` enum value assigned to every newly inserted Resume_Record, indicating the file has been received but not yet analyzed.

---

## User Identity Design Clarification

The `users` table has the following structure (confirmed by migration `0000_right_marvex.sql`):

```
users.id       — TEXT PRIMARY KEY  — stores the Clerk User ID (e.g. "user_2abc…")
users.clerk_id — TEXT UNIQUE NOT NULL — stores the same Clerk User ID (redundant duplicate)
```

The Clerk webhook handler (`app/api/webhooks/clerk/route.ts`) explicitly sets both columns to the same value on every `user.created` event:

```
id:      data.id   // Clerk userId
clerkId: data.id   // identical value
```

**Consequence for the Upload_API:** `auth().userId` from `@clerk/nextjs/server` returns the Clerk User ID, which is directly usable as the value for `resumes.user_id` (the foreign key referencing `users.id`). No additional database lookup is needed to resolve an internal identifier — `users.id` IS the Clerk User ID. This is a deliberate design simplification, not a schema error.

## Requirements

---

### Requirement 1: Authentication Enforcement

**User Story:** As a product owner, I want every upload attempt to be authenticated, so that only registered users can store resumes and consume storage resources.

#### Acceptance Criteria

1. WHEN a request is received at `POST /api/resumes/upload`, THE Upload_API SHALL verify the caller's identity by invoking `auth()` from `@clerk/nextjs/server` on the server side.
2. IF the `auth()` call returns a null or absent `userId`, THEN THE Upload_API SHALL return an HTTP 401 response with a JSON body containing the field `"error"` set to a human-readable unauthenticated message.
3. THE Upload_API SHALL retrieve the Clerk_User_ID exclusively from the server-side `auth()` call and SHALL NOT accept any user identity value supplied in the request body or headers by the client.
4. WHILE a valid `userId` is present in the `auth()` result, THE Upload_API SHALL proceed to file validation without performing any additional database lookup to resolve the user's primary key.

---

### Requirement 2: File Validation

**User Story:** As a user, I want the system to clearly reject files that are the wrong type or too large, so that I receive immediate feedback and can correct my selection before wasting time.

#### Acceptance Criteria

1. WHEN a Resume_File is received, THE Upload_API SHALL validate that the file's MIME_Type is exactly `"application/pdf"` or `"application/vnd.openxmlformats-officedocument.wordprocessingml.document"`.
2. WHEN a Resume_File is received, THE Upload_API SHALL validate that the file's original name ends with the extension `.pdf` or `.docx` (case-insensitive).
3. WHEN a Resume_File is received, THE Upload_API SHALL validate that the file's size in bytes does not exceed 5,242,880 bytes (5 MB).
4. IF the MIME_Type validation fails, THEN THE Upload_API SHALL return an HTTP 422 response with a JSON body containing an `"error"` field that identifies the received MIME type and lists the accepted types.
5. IF the file extension validation fails, THEN THE Upload_API SHALL return an HTTP 422 response with a JSON body containing an `"error"` field that identifies the received extension and lists the accepted extensions.
6. IF the file size exceeds 5,242,880 bytes, THEN THE Upload_API SHALL return an HTTP 422 response with a JSON body containing an `"error"` field that states the maximum allowed size and the actual received size in bytes.
7. THE Upload_API SHALL perform MIME_Type validation, extension validation, and size validation before initiating any storage or database operation.

---

### Requirement 3: Resume ID Generation

**User Story:** As an engineer, I want each uploaded resume to receive a unique, collision-safe identifier, so that storage paths and database records can be reliably associated without conflicts.

#### Acceptance Criteria

1. WHEN a Resume_File passes all validation checks, THE Upload_API SHALL generate a Resume_ID by calling `crypto.randomUUID()` from the Node.js built-in `crypto` module.
2. THE Upload_API SHALL generate the Resume_ID before constructing the Storage_Path or inserting any database record.
3. THE Upload_API SHALL NOT introduce any new npm dependency (such as `nanoid`) for ID generation; `crypto.randomUUID()` is the chosen mechanism.

---

### Requirement 4: Supabase Storage Upload

**User Story:** As a user, I want my resume file to be stored securely so that it can be retrieved later for analysis.

#### Acceptance Criteria

1. WHEN a Resume_File passes validation and a Resume_ID has been generated, THE Upload_API SHALL upload the file to the Supabase Storage bucket named `"resumes"` using the Service_Client.
2. THE Upload_API SHALL construct the Storage_Path as `{clerkUserId}/{resumeId}.{ext}`, where `{ext}` is `"pdf"` for PDF files and `"docx"` for DOCX files, matching the convention already implemented in `lib/supabase/storage.ts`.
3. THE Upload_API SHALL use the existing `uploadResume(userId, resumeId, buffer, mimeType)` function from `lib/supabase/storage.ts` to perform the upload.
4. THE Upload_API SHALL use the Service_Client (initialized with `SUPABASE_SERVICE_ROLE_KEY`) for all storage operations and SHALL NOT use the Anon_Client for this purpose.
5. IF the `"resumes"` bucket does not exist in Supabase Storage, THEN THE Upload_API SHALL return an HTTP 500 response with a JSON body containing an `"error"` field that instructs the operator to create the `"resumes"` bucket manually in the Supabase dashboard.
6. IF the storage upload fails for any other reason, THEN THE Upload_API SHALL return an HTTP 500 response with a JSON body containing an `"error"` field describing the failure, and SHALL NOT proceed to the database insert step.
7. WHEN the storage upload succeeds, THE Upload_API SHALL retain the returned Storage_Path and Signed_URL for use in the subsequent database insert.

---

### Requirement 5: Database Record Insertion

**User Story:** As a user, I want a database record to be created for each successfully uploaded resume, so that I can track my uploads and their processing status.

#### Acceptance Criteria

1. WHEN the storage upload succeeds, THE Upload_API SHALL insert a Resume_Record into the `resumes` table using the Drizzle ORM `db` instance from `db/index.ts`.
2. THE Upload_API SHALL populate the Resume_Record with the following required fields: `id` (the generated Resume_ID), `userId` (the Clerk_User_ID from `auth()`), `originalName` (the file's original filename as provided in FormData), `storagePath` (the Storage_Path returned by the storage upload), `storageUrl` (the Signed_URL returned by the storage upload), `mimeType` (the validated MIME_Type), `sizeBytes` (the file size in bytes), and `status` set to `"pending"`.
3. THE Upload_API SHALL store the Clerk_User_ID directly in `resumes.userId` without performing any intermediate database lookup, as `users.id` IS the Clerk_User_ID.
4. IF the database insert fails, THEN THE Upload_API SHALL attempt to delete the already-uploaded file from Supabase Storage using the existing `deleteResume(path)` function from `lib/supabase/storage.ts` as a best-effort cleanup operation.
5. IF the database insert fails, THEN THE Upload_API SHALL return an HTTP 500 response with a JSON body containing an `"error"` field describing the failure, regardless of whether the cleanup deletion succeeded.
6. THE Upload_API SHALL NOT insert a Resume_Record before confirming that the storage upload has succeeded.

---

### Requirement 6: API Response Contract

**User Story:** As a client developer, I want the upload API to return consistent, predictable responses, so that the Upload_UI can reliably handle success and error states.

#### Acceptance Criteria

1. WHEN both the storage upload and database insert succeed, THE Upload_API SHALL return an HTTP 201 response with a JSON body containing: `"resumeId"` (the generated Resume_ID as a string), `"storagePath"` (the Storage_Path as a string), and `"status"` set to the string `"pending"`.
2. WHEN a request is rejected due to missing authentication, THE Upload_API SHALL return an HTTP 401 response with a JSON body containing an `"error"` field with a human-readable message.
3. WHEN a request is rejected due to invalid file type or file size, THE Upload_API SHALL return an HTTP 422 response with a JSON body containing an `"error"` field with a human-readable message describing the specific validation failure.
4. WHEN a request fails due to a storage or database error, THE Upload_API SHALL return an HTTP 500 response with a JSON body containing an `"error"` field with a human-readable message.
5. THE Upload_API SHALL return all error responses as JSON with a `Content-Type` of `"application/json"`.
6. THE Upload_API SHALL NOT expose internal stack traces, Supabase error objects, or raw database error messages in any response body returned to the client.

---

### Requirement 7: Upload UI — File Selection

**User Story:** As a user, I want an intuitive drag-and-drop interface for selecting my resume, so that I can upload my file without friction.

#### Acceptance Criteria

1. THE Upload_UI SHALL render as a Client Component (with `"use client"` directive) replacing the static Server Component at `app/(dashboard)/upload/page.tsx`.
2. THE Upload_UI SHALL provide a drag-and-drop drop zone that accepts files dragged from the user's file system.
3. THE Upload_UI SHALL provide a click-to-browse interaction that opens the native OS file picker when the drop zone or "Browse Files" button is activated.
4. WHEN a file is selected via drag-and-drop or file picker, THE Upload_UI SHALL display the selected file's name to the user before the upload begins.
5. THE Upload_UI SHALL preserve the existing visual design tokens from the static placeholder, including `border-border`, `bg-surface`, `bg-surface-subtle`, `text-foreground`, `text-foreground-muted`, `text-foreground-subtle`, `bg-primary`, `text-primary`, `ring-primary-muted`, and `rounded-2xl` border radius conventions.

---

### Requirement 8: Upload UI — Upload Lifecycle

**User Story:** As a user, I want clear feedback during and after the upload, so that I know whether my file was accepted or if I need to take action.

#### Acceptance Criteria

1. WHEN the user initiates an upload, THE Upload_UI SHALL send the selected file as a `multipart/form-data` POST request to `POST /api/resumes/upload` with the file attached under the field name `"file"`.
2. WHILE an upload is in progress, THE Upload_UI SHALL display a loading indicator (spinner or progress state) to communicate that the operation is ongoing.
3. WHILE an upload is in progress, THE Upload_UI SHALL disable the upload button and drop zone to prevent duplicate submission of the same or a different file.
4. WHEN the Upload_API returns an HTTP 201 response, THE Upload_UI SHALL display a success state that includes the uploaded file's name and a confirmation message.
5. WHEN the Upload_API returns any non-201 HTTP response, THE Upload_UI SHALL display an error state that includes the human-readable `"error"` string from the response JSON body.
6. WHEN a file fails client-side size or type pre-validation (see Requirement 9), THE Upload_UI SHALL display a descriptive error message without sending a request to the Upload_API.
7. IF an upload has completed (successfully or with an error), THE Upload_UI SHALL allow the user to dismiss the result and select a new file for another upload attempt.

---

### Requirement 9: Upload UI — Client-Side Pre-validation

**User Story:** As a user, I want immediate feedback if I select a file that is clearly wrong, so that I do not have to wait for a server round-trip to learn of a basic error.

#### Acceptance Criteria

1. WHEN a file is selected, THE Upload_UI SHALL verify that the file's `type` property matches `"application/pdf"` or `"application/vnd.openxmlformats-officedocument.wordprocessingml.document"` before initiating the upload.
2. WHEN a file is selected, THE Upload_UI SHALL verify that the file's `name` property ends with `.pdf` or `.docx` (case-insensitive) before initiating the upload.
3. WHEN a file is selected, THE Upload_UI SHALL verify that the file's `size` property does not exceed 5,242,880 bytes before initiating the upload.
4. IF client-side pre-validation fails, THEN THE Upload_UI SHALL display an inline error message describing the specific failure and SHALL NOT send the file to the Upload_API.
5. THE Upload_UI SHALL treat client-side pre-validation as a convenience check only; server-side validation (Requirement 2) remains the authoritative enforcement layer.

---

### Requirement 10: Security and Credential Isolation

**User Story:** As a security-conscious engineer, I want all sensitive operations to remain server-side, so that Supabase credentials and Clerk session verification cannot be compromised by client-side code.

#### Acceptance Criteria

1. THE Upload_API SHALL perform all Clerk authentication checks, Supabase storage writes, and database inserts exclusively within the Next.js Route Handler, which runs in the Node.js server environment.
2. THE Upload_UI SHALL communicate with the Upload_API only via the standard `fetch` API sending FormData; it SHALL NOT import or invoke any Supabase server client, service client, or Drizzle ORM instance.
3. THE Upload_API SHALL access `SUPABASE_SERVICE_ROLE_KEY` only through the `createServiceClient()` function from `lib/supabase/server.ts`, which is a server-only module.
4. THE Upload_UI SHALL NOT access `SUPABASE_SERVICE_ROLE_KEY` directly or indirectly; the environment variable MUST remain absent from all client-side bundles.
5. THE Upload_API SHALL NOT introduce any new environment variables beyond those already defined in the project (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `CLERK_SECRET_KEY`, and `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`).

---

### Requirement 11: Error Atomicity and Cleanup

**User Story:** As an engineer, I want the system to avoid orphaned storage objects when a database insert fails, so that storage usage remains consistent with database records.

#### Acceptance Criteria

1. IF the database insert fails after a successful storage upload, THEN THE Upload_API SHALL call `deleteResume(storagePath)` from `lib/supabase/storage.ts` to attempt removal of the orphaned file.
2. THE Upload_API SHALL treat the storage cleanup as a best-effort operation; if `deleteResume` itself fails or throws, THE Upload_API SHALL log the failure and proceed to return the HTTP 500 error response without rethrowing.
3. THE Upload_API SHALL NOT insert a Resume_Record if the storage upload has not returned a successful result, preserving the invariant that every Resume_Record has a corresponding file in storage.
4. IF a storage upload fails, THEN THE Upload_API SHALL return an HTTP 500 response immediately without attempting any database write.

---

## Out of Scope

The following capabilities are explicitly excluded from this feature and must not be designed or implemented as part of the Resume Upload Pipeline:

- **Text extraction**: Parsing the content of PDF or DOCX files to extract plain text.
- **AI / LLM analysis**: Sending resume content to OpenAI or any other language model for scoring or feedback.
- **ATS scoring**: Computing applicant-tracking-system compatibility scores.
- **Resume listing**: Endpoints or UI to display a user's previously uploaded resumes.
- **Resume deletion**: Endpoints or UI to remove a resume from storage or the database.
- **Resume retrieval**: Endpoints or UI to download or preview an uploaded resume.
- **Job description targeting**: Populating or editing the `target_job_title` or `target_job_description` fields at upload time.
- **Schema migrations**: Creating new database tables, columns, or enum values. The existing schema is used as-is.
- **New Supabase buckets**: The `"resumes"` bucket must already exist; bucket creation is not part of this feature.
- **Rate limiting**: Throttling upload requests per user or per IP address.
- **Virus or malware scanning**: Inspecting uploaded file content for malicious payloads.
- **Multi-file upload**: Uploading more than one resume in a single request.
